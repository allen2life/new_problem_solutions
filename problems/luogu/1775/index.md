### 题意

给出一排石子，每次只能合并相邻的两堆，代价等于这两堆石子的质量之和。多次合并后要把所有石子变成一堆，求最小总代价。

题目的关键限制是“只能合并相邻两堆”。这意味着任意时刻形成的新石子堆，都对应原序列中的一个连续区间。

### 思路

最直接的想法是按照题意暴力模拟：每一步都枚举当前要合并的相邻两堆，递归搜索所有可能的合并顺序。

这个暴力写法是可信的，因为它完整枚举了所有合法操作顺序，但复杂度是指数级，只能拿来做小数据验证。

真正的优化点在于换一种观察角度。对于任意区间 `[l, r]`，如果它最终被合并成一堆，那么最后一步一定是把某个 `[l, k]` 和 `[k+1, r]` 合并起来。也就是说，我们不必关心“前面每一步怎么合并”，只需要关心“最后一次在哪里断开”。

设 `dp[l][r]` 表示把区间 `[l, r]` 合并成一堆的最小代价，那么有转移：

`dp[l][r] = min(dp[l][k] + dp[k+1][r] + sum(l, r))`

其中 `l <= k < r`，`sum(l, r)` 表示区间总质量。为了在转移时快速求区间和，用前缀和数组维护即可。

这张表展示 `dp[l][r]` 在几个典型区间上的含义：

| 状态 | 表示什么 |
| --- | --- |
| `dp[2][2]` | 第 2 堆单独存在，不需要合并，代价是 `0` |
| `dp[2][3]` | 把第 2 堆到第 3 堆合成一堆的最小代价 |
| `dp[1][4]` | 把整段石子全部合成一堆的最小代价，也就是最终答案 |

读这张表时，重点是把 `dp[l][r]` 理解成“一个连续区间的子问题”。一旦状态这样定义，最后一次合并就天然对应一个断点 `k`。于是三重循环按区间长度递推，就能覆盖所有可能的最优方案。

实现时，`brute.cpp` 枚举的是当前操作顺序；`main.cpp` 枚举的是最后一次合并的断点。这就是两份代码的本质差别。

### 复杂度

区间 DP 需要枚举区间长度、左端点和断点，所以时间复杂度是 `O(n^3)`，空间复杂度是 `O(n^2)`。

### 总结

这题的核心不是“怎么模拟合并”，而是识别出“相邻合并”对应连续区间结构。把思路从枚举操作顺序改成枚举最后一次合并位置后，就能自然得到区间 DP。
