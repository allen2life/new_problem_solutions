### 题意

有 `N` 个球员，球一开始在 `1` 号球员脚下，目标是把球送到 `N` 号球员所在的位置。

操作分三类：

- 控球时可以运球 1 米，代价 `C`
- 控球时可以朝上下左右某个方向踢 `p` 米，代价 `A * p + B`，踢完后球员失去控球
- 没控球的球员可以自己走 1 米，代价也是 `C`；如果他和球在同一格且当前没人控球，就能接球

要求最小化所有球员疲劳度之和。

### 思路

先看一个小数据可以直接验证的版本：

@include-code(./brute.cpp, cpp)

朴素版把状态写成两层：

- `FREE(x, y)`：球在 `(x, y)`，当前没人控球
- `CONTROL(x, y)`：球在 `(x, y)`，当前有人控球

然后在 `CONTROL` 状态里显式枚举同行同列的所有踢球落点。这个做法很好理解，但一旦格子数到 `500 x 500`，每次都枚举整行整列就太慢了。

#### 关键观察 1：不用跟踪“到底是哪位球员”

如果球现在静止在某个格子 `(x, y)`，我们只关心：

- 把“某个球员”调到这里来接球，最少要花多少代价

这个值记成 `need[x][y]`。因为所有球员走 1 米的代价都是 `C`，所以它就是：

- `need[x][y] = 最近球员到 (x, y) 的曼哈顿距离 * C`

更重要的是，这个值和历史无关。  
原因是：如果某个球员之前已经走到别的地方了，再让他去 `(x, y)`，总代价一定不小于“直接从他的初始位置走到 `(x, y)`”。这是曼哈顿距离的三角不等式。

所以一旦球处于“无人控球”的状态，我们完全不必记录哪位球员以前做过什么，只要付出 `need[x][y]`，就能认为“有一位最便宜的球员来到这里接球”。

#### 关键观察 2：把踢球拆成飞行状态

主程序不再从一个控球格子直接连向整行整列，而是把“球正在朝某个方向飞”也看成状态：

```mermaid
flowchart LR
  F["FREE(x,y) 空球"] -->|need[x][y]| C["CONTROL(x,y) 控球"]
  C -->|+C| C2["相邻格子, 继续控球"]
  C -->|+A+B| K["开始朝某方向飞 1 米"]
  K -->|+A| K2["继续朝同方向飞"]
  K -->|0| F2["在当前位置停下, 变成空球"]
```

这样一来，踢球就从“一个点向整条线连边”，变成了“沿一个方向一步一步走”。  
总代价完全一样，但每个状态的出边数变成常数了。

#### 最终建图

我们只在球场内部 `0..H, 0..W` 这些格子上建图。最优解没有必要把球和球员带到场外，因为：

- 两个场内格子之间，走场内的最短路不会比出界更差
- 同一行或同一列的两个场内格子之间，踢球直线段也一直在场内
- 先把球带到场外再带回来，只会额外增加距离

于是状态一共 6 层：

- `FREE`
- `CONTROL`
- `KICK_NORTH`
- `KICK_SOUTH`
- `KICK_WEST`
- `KICK_EAST`

边如下：

1. `FREE -> CONTROL`，代价 `need[x][y]`
2. `CONTROL -> 相邻 CONTROL`，代价 `C`
3. `CONTROL -> 相邻 KICK_*`，代价 `A + B`，表示踢球至少先飞 1 米
4. `KICK_* -> 同方向下一格 KICK_*`，代价 `A`
5. `KICK_* -> FREE`，代价 `0`，表示球停在这里

起点是 `1` 号球员一开始就控球，所以：

- `CONTROL(S1, T1) = 0`

终点是球到达 `N` 号球员所在格子即可，所以答案取：

- `min(FREE(SN, TN), CONTROL(SN, TN))`

### 复杂度

先用多源 BFS 预处理 `need[x][y]`，复杂度 `O(HW + N)`。

最短路里一共是 `6 * (H + 1) * (W + 1)` 个状态，每个状态只有常数条出边，所以 Dijkstra 的复杂度是：

- `O(HW log(HW))`

空间复杂度：

- `O(HW)`

### 总结

这题最难的地方不是写 Dijkstra，而是先把题目里的“人、球、控球权”理顺。

一旦想清楚两件事：

1. 空球状态下，只需要知道最近球员来接球的最小代价
2. 踢球可以拆成 4 个方向飞行状态

整题就变成一张标准状态图最短路。
